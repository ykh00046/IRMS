## 0. 현재 상태 진단

### 강점 (지킬 것)
- **기능 완결**: 배합 실적(DHR)·레시피 관리(개정 체인)·점도 분석·근태가 실제 현장
  워크플로(구 Program-estimation)를 대체. 등록→기록→출력→분석 루프가 닫혀 있음.
- **품질 기반**: 테스트 271개, 커밋마다 전체 테스트 + 브라우저 QA 관행. 감사 로그,
  세션 토큰 회전, CSRF, 내부망 제한 등 보안 기본기 갖춤.
- **운영 자동화**: run_auto.bat(git pull 감시 + 자동 재시작), 업데이트 전 DB 백업,
  트레이 앱(근태·점도 알림 + 저울 연동) 배포 체계.
- **코드 위생**: 2026-07-07 데드코드 일괄 제거로 코드·문서·화면이 현행과 일치.

### 약점 (보완 대상 — 검토에서 확인된 사실 기반)
1. 백업이 **업데이트 시점에만** 수행 — 코드 변경이 없는 기간엔 백업 공백.
2. serve.py 가 포트 점유 시 **크래시 루프** (update_and_run.bat 에만 포트 정리 있음).
3. 의존성이 하한(>=)만 고정 + 매 기동 pip install → **무통제 업그레이드 가능**.
4. 라우트 테스트 공백: 근태 로그인(422 사고 이력 파일!), 점도 등록/삭제, 결재,
   서명/시트백업 관리, blend bulk·삭제.
5. 이용자 **이름 변경 시 과거 배합 기록과 연결 단절** (+삭제 안전장치 우회 가능).
6. 날짜 기준 혼용: 대시보드/리마인더는 로컬, 점도 등록일은 UTC → 자정 부근
   "오늘 미입력" 판정이 하루 밀릴 수 있음.
7. admin/admin 폴백 계정 정책 미결정 (화면 없는 users CRUD API 존치 여부 포함).
8. 외부 접속(Cloudflare Tunnel)은 구축 완료·sign-off 대기 — 단, 프록시 뒤에서는
   rate-limit·내부망 판정이 IP 를 못 보므로 **토큰 모드 전제 재점검 필요**.

## 1. 발전 방향 3축

| 축 | 목표 | 근거 |
|---|---|---|
| **① 운영 안정성** | "데이터를 잃지 않는다, 서버는 스스로 복구한다" | 단일 SQLite + 단일 운영 PC 구조. 기록(DHR)은 인허가 문서라 유실 = 사고 |
| **② 품질 완결성** | "바꿔도 깨지지 않는다" | 기능 추가 속도가 빠름 → 회귀 방지망(라우트 테스트)이 속도를 지탱 |
| **③ 데이터 활용** | "쌓인 기록이 판단을 돕는다" | 배합·점도 데이터가 축적 중 — 분석·리포트로 환원해야 입력 동기 유지 |

## 2. 플랜

### Phase 1 — 운영 안정성 (단기, 1~2주) ★최우선

- **P1-1. 일일 자동 DB 백업 + 보존 정책**
  serve.py 감시 루프에 일 1회 백업 추가(기존 backup_db 재사용), `backups/` 보존
  N=30일 초과분 자동 삭제, 복구 절차를 CLAUDE.md/README 에 3줄로 문서화.
  선택: 백업본을 두 번째 위치(외장/네트워크 폴더)로 복사하는 옵션.
- **P1-2. 포트 충돌 자동 해제**
  serve.py 기동 전 9000 포트 점유 프로세스 정리(update_and_run.bat 로직 이식)
  → 크래시 루프 제거.
- **P1-3. 의존성 잠금**
  운영 검증 버전으로 `requirements-lock.txt`(pip freeze) 생성, serve.py 는 lock
  파일로 설치. 업그레이드는 개발 PC에서 테스트 후 lock 갱신 커밋으로만.
- **P1-4. admin/admin 폴백 정책 확정** (결정 필요 — §3)

### Phase 2 — 품질 완결성 (중기, 2~4주)

- **P2-1. 라우트 테스트 공백 해소** (우선순위순)
  ① 근태 로그인/비번변경/초기화 — future annotations 422 **사고 이력이 있는 파일**
  ② 점도 측정 등록/삭제 ③ blend bulk·DELETE ④ 서명/시트백업 관리 API.
  목표: "쓰기 라우트 = 최소 1개 라우트 테스트" 원칙 확립.
- **P2-2. 이름 변경 ↔ 기록 동기화**
  rename 시 blend_records.worker 동반 갱신(오타 정정이 목적이므로 타당) +
  감사 로그. has_blend_records 안전장치 정상화.
- **P2-3. 날짜 기준 통일**
  "오늘" 판정을 로컬 날짜 하나로 통일(현장 운영은 KST 단일 사이트).
  utc_now_text 저장은 유지, **판정·표시만 로컬 기준** 명문화.
- **P2-4. 점진 표준화** (기능 작업에 얹어서)
  에러 detail 한글 문구 통일, 라우터 권한 선언은 `dependencies=[...]` 패턴으로 수렴.

### Phase 3 — 데이터 활용 (중기~장기, 1~2개월)

- **P3-1. 점도 운영 데이터 완성**
  과거 점도 엑셀의 운영 DB 적재(스크립트는 준비됨 — **운영 PC에서 실행**),
  이후 추세·이상 판정이 실데이터 기반으로 동작하는지 현장 검증 1주.
- **P3-2. 월간 리포트 자동화**
  배합 분석(insight)·점도 추세를 월 단위 Excel/PDF 로 뽑는 "월간 리포트" 버튼.
  (수요 확인 후: 시트 백업처럼 자동 생성·보관까지)
- **P3-3. 배합 데이터 심화** (수요 기반 — 현장 요청 있을 때만)
  제품별 배치 간 편차 추이, 자재 LOT 별 점도 상관 등. 새 화면보다 insight 확장으로.

### 백로그 (착수 미정 — 운영자 지시 대기)

- **B-1. 저울 연동 시 계량값 수동 입력 잠금** — ✅ 완료 (2026-07-09)
  저울 감지 시 실제량 칸 readonly, 미감지 시 자동 해제. '수동 입력' 토글로
  ①켤 때 확인+배너 안내 ②blend_records.manual_entry 저장 ③기록 목록 ⚠·상세
  '⚠ 수동 입력' 배지 + 감사 로그. 294 passed, 브라우저 QA 완료.

### Phase 4 — 장기 과제 (분기 단위 재평가)

- **P4-1. 데이터 성장 대비**: DB 크기·조회 속도 반기 점검, 임계 도달 시 연 단위
  아카이빙(오래된 기록 분리 보관 + 조회 시 통합) 설계.
- **P4-2. 외부 접속 상시화 여부**: sign-off 후 운영 시 프록시 뒤 rate-limit
  (X-Forwarded-For 미신뢰 전제) 재설계 + 접속 감사 강화.
- **P4-3. 인증 확장성**: 책임자 수가 늘거나 외부 접속이 상시화되면 2단계 인증
  (근태처럼 잠금 정책은 이미 있음) 검토.
