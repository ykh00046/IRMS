# 🎨 잉크 레시피 관리 시스템(IRMS) 고도화 구축 플랜

본 문서는 현장 작업 효율 극대화와 데이터 무결성 확보를 위한 **사내 잉크 레시피 관리 시스템(IRMS)**의 최종 기획 사양을 담고 있습니다.

---

## 1. 프로젝트 개요

**목적**: 수기/엑셀 관리 레시피의 디지털 전환을 통한 제조 오류 방지 및 자재 추적성 확보

**핵심 컨셉**:

- **Workflow-Centric**: 현장 작업 동선에 맞춘 가로형 대시보드
- **Contamination-Free**: 안료 교체 시 오염을 방지하는 '컬러 포커스(Focus Mode)' 기능
- **Zero-Entry**: 엑셀 붙여넣기를 통한 입력 자동화

---

## 2. 사용자 권한 체계 (총 8명)

**관리자 (2명)**:

- 원재료 마스터 DB 등록/수정/삭제
- 사용자 계정 관리 및 전체 레시피 이력 관리

**일반 사용자 (6명)**:

- 레시피 등록 (엑셀 붙여넣기 활용)
- 제조 현장 대시보드 조회 및 완료 체크
- 기간별 원재료 소요량 조회

### 2.1 인증 방식

| 항목          | PoC (Phase 1)               | 정식 운영 (Phase 2) |
| ------------- | --------------------------- | ------------------- |
| 인증 방식     | ID/PW 기반 로그인           | SSO 연동 검토       |
| 비밀번호 정책 | 최소 8자, 관리자 초기 발급  | 주기적 변경 강제    |
| 세션 관리     | 8시간 자동 만료             | 동일                |
| 공유 계정     | **사용 금지** (추적성 보장) | 동일                |

## 3. 화면별 상세 사양

### [화면 1] 제조 현장 대시보드 (Work Mode)

현장 작업자가 레시피를 확인하고 투입량을 체크하는 핵심 화면입니다.

- **UI 구조**: 엑셀 시트와 동일한 가로형 테이블(Horizontal Table) 레이아웃
- **정렬 로직**: 등록된 순서대로 상단부터 하단으로 배치 (Sequential Order)

#### 집중 계량 모드 (Color Focus Mode)

- **기능**: 상단 'Black, Red, Blue, Yellow' 버튼 클릭 시 해당 색상 원료 컬럼만 노출
- **목적**: 특정 안료 소분 시 장갑 오염 방지 및 작업 집중도 향상 (나머지 컬럼은 숨김 처리)

#### 완료 처리

- 우측 '완료' 버튼 클릭 시 해당 행이 화면에서 즉시 사라지며 통계 데이터로 이관
- **2단 확인 필수**: 오작동 방지를 위해 확인 팝업 또는 길게 누르기 적용

#### 검색/필터 (추가)

- 제품명/잉크명 검색 기능
- 날짜 기반 필터링

#### 실시간 동기화

- 다수 사용자 동시 작업 시 완료 상태 실시간 반영 (WebSocket 또는 Server-Sent Events)

### [화면 2] 레시피 등록 및 조회 (Management)

- **Smart Import**: 엑셀의 데이터 영역을 복사하여 붙여넣으면 헤더를 분석해 자동 매핑
- **미리보기 단계**: 붙여넣기 → Preview 확인 → 저장 (3단계 프로세스)

#### 유효성 검사 (Validation)

- 마스터 DB에 없는 원재료 포함 시 경고 및 등록 차단
- 숫자 데이터(중량)와 문자 데이터(횟수/코드) 자동 구분 인식

#### 다중 필터 조회

- 등록자별, 기간별, 제품명별 미완료/완료 레시피 추적

### [화면 3] 원재료 소비 통계 (Insight)

#### 데이터 집계 이원화

- **수치형(Weight)**: 완료된 레시피의 투입량을 누적 합산 (단위: Kg)
- **문자형(Count)**: 코드로 기록된 품목(예: PB-APB 등)은 사용 횟수로 카운트

#### Report

- 기간 설정 후 집계된 데이터를 엑셀 파일로 추출하여 재고 발주 자료로 활용
- 기간 대비 증감률, 전월 대비 비교 분석 포함

## 4. 데이터 관리 정책

| 구분   | 관리 항목                  | 데이터 타입                  | 비고             |
| ------ | -------------------------- | ---------------------------- | ---------------- |
| 원재료 | 명칭, 단위, 색상그룹       | String, Enum, Color          | 마스터 DB 기반   |
| 레시피 | 제품명, 잉크명, 투입데이터 | String, 정규화(recipe_items) | 엑셀 파싱 데이터 |
| 로그   | 등록자, 완료시각, 상태     | String, Timestamp            | 시스템 자동 생성 |

> 레시피 투입데이터는 `recipe_items` 테이블로 정규화하여 집계 쿼리, 인덱싱, 정합성을 보장한다.

## 5. 기술적 특징 및 기대 효과

### 🛠 기술적 특징

- **가로형 유동 헤더**: 등록된 레시피에 포함된 원재료에 따라 테이블 컬럼이 유동적으로 구성됨
- **Sticky UI**: 제품명과 완료 버튼을 좌우에 고정하여 가로 스크롤 시에도 식별 용이
- **Focus Mode Logic**: 필터링 시 데이터가 있는 행뿐만 아니라 필수 컬럼 자체를 필터링하는 현장 맞춤형 로직

### 📈 기대 효과

- **작업 시간 단축**: 컬러별 모아보기 기능을 통해 안료 교체 및 소분 동선 최적화
- **오류 감소**: 엑셀 자동 파싱 및 원재료 검증으로 오기입 근절
- **재고 예측 정밀도 향상**: 소모량(중량+횟수)의 자동 산출로 정확한 원재료 관리 가능

- **재고 연동**: 소모량 발생 시 원재료 마스터의 가상 재고에서 자동 차감 및 알림

## 7. 제공/운영 방식 (Option A: 로컬 서버 PoC 운영안)

본 시스템은 사내 사용을 전제로 하며, 1차 도입은 **Option A(로컬 서버 PoC)** 방식으로 빠르게 검증 후, 필요 시 중앙 서버(정식 운영)로 확장 가능한 구조를 권장한다.  
(기존 8명 사용자 운영 및 Work/Management/Insight 3화면 구조는 유지)

### 7.1 Option A 개요

| 항목           | 내용                                                                                                |
| -------------- | --------------------------------------------------------------------------------------------------- |
| 운영 형태      | 사내 네트워크 내 **단일 PC(Host)** 에 서버 실행, 사용자들은 브라우저로 접속                         |
| 목표           | **현장 가치 검증(Work Mode 집중 계량 + Zero-Entry 등록 + 소모량 통계)** 을 2~4주 내 실사용으로 확인 |
| 권장 사용자 수 | 동시 1~8명 (초기 운영 가능), 사용 증가 시 중앙 서버 확장 검토                                       |
| DB             | 1차: SQLite(로컬 파일) / 2차 확장 시 Postgres로 마이그레이션 가능                                   |
| 접근           | 사내 폐쇄망 IP 접근 (예: http://<HOST-IP>:<PORT>)                                                   |

> Option A는 "빨리 쓰면서 문제를 드러내고(입력 규칙/무결성/로그/현장 UX), 정식 운영 설계를 고도화"하기 위한 단계이다.

### 7.2 권장 설치/운영 절차 (PoC 표준)

1. **Host PC 선정**

- 사내 폐쇄망 고정 IP 또는 DHCP 예약 권장
- 전원/절전 정책: 업무시간 상시 켜짐(또는 최소 작업시간 유지)

2. **서비스 실행**

- 방식 A: Docker(권장)로 서버 실행
- 방식 B: Python/Node 런타임으로 직접 실행(간단하지만 환경 이슈 발생 가능)

3. **접속 안내**

- 사용자들은 브라우저 즐겨찾기 고정
- 작업장 터치 단말 사용 시 전체화면(키오스크/전체화면) 권장

4. **운영 책임(R&R)**
   | 역할 | 담당 | 책임 |
   |---|---|---|
   | PoC 운영 관리자 | 관리자 1~2명 | 마스터 DB 관리, 사용자 계정 관리, 백업 확인 |
   | 현장 운영 | 일반 사용자 | Work Mode 작업/완료 처리, 특이사항 메모 |
   | 시스템 관리(필요 시) | IT/설비 | Host PC 보안/네트워크/장애 대응(최소) |

### 7.3 백업/복구 정책 (Option A 필수)

- **DB 백업 주기**: 1일 1회(업무 종료 후) 자동 백업 권장
- **보관 기간**: 최근 7일(최소) / 30일(권장)
- **백업 대상**
  - DB 파일(SQLite)
  - 업로드/첨부 파일(사용 시)
  - 시스템 로그 파일(사용 시)
- **복구 원칙**
  - 장애 발생 시: 최신 백업 DB로 복구 후 서비스 재기동
  - 복구 이력은 관리자 로그에 "복구 시각/사유/복구본" 기록
- **Host PC 장애 대응**
  - Docker 이미지 + DB 백업을 대체 PC에 복사하여 빠르게 전환
  - 전환 절차 문서 사전 준비 및 관리자 교육

### 7.4 보안/접근 통제 (Option A 최소 기준)

- 접속 범위: 사내망(폐쇄망) 내부로 제한
- 계정/권한: 역할(RBAC) 기반(관리자/일반 사용자) 유지
- 데이터 보호: Host PC 접근 권한 제한(공용 계정 사용 금지)
- 감사/추적: 핵심 이벤트(등록/수정/완료/취소/마스터 변경)는 Audit Log에 기록(8장 참조)

### 7.5 Option A의 한계 및 확장 트리거

| 트리거(조건)                            | 의미                  | 권장 조치                     |
| --------------------------------------- | --------------------- | ----------------------------- |
| Host PC 장애/전원 문제로 업무 영향 빈번 | 가용성 부족           | 중앙 서버(사내 VM) 전환 검토  |
| 동시 사용자 증가로 지연/충돌            | 성능/동시성 한계      | DB Postgres 전환 + 서버 분리  |
| 재고 연동/바코드/알림이 본격화          | 데이터/연동 요구 증가 | Phase2 착수 시 중앙 운영 권장 |
| 감사/증빙 요구 강화                     | 로그/보존 요구 증가   | 로그 보관 정책 강화 + 중앙화  |

### 7.6 업데이트/배포 정책 (추가)

- PoC 기간 중 시스템 업데이트(버그 수정/기능 추가) 시 배포 절차:
  - 업무 종료 후 백업 → 업데이트 적용 → 검증 → 배포
  - 필요 시 DB 마이그레이션 스크립트 포함
- **Health Check**: 서버 상태 확인용 엔드포인트(`/health`) 제공
- 디스크 사용량 임계치(80%) 도달 시 관리자 알림

## 8. 레시피 상태/수정/감사로그 정책 (데이터 무결성 강화)

기존 기획의 "완료 처리 시 Work 화면에서 사라지고 통계로 이관" UX는 유지하되, 데이터 관점에서는 **상태 전이 + 완료 잠금(LOCK) + 감사로그(Audit Log)** 를 표준으로 적용한다.  
(Work/Management/Insight 구조 및 완료 처리 개념 유지)

### 8.1 레시피 상태(State) 정의

| 상태        |                                 의미 | Work 노출 | 수정 가능 | 비고                     |
| ----------- | -----------------------------------: | --------: | --------: | ------------------------ |
| Draft       |                등록 중(검증 전/임시) |         X |         O | 내부 임시 저장용(선택)   |
| Pending     |                   현장 대기(작업 전) |         O |   O(제한) | 기본 Work 대상           |
| In-Progress | 작업 시작(작업자 할당/시작시각 기록) |         O |      제한 | 책임 추적성 강화         |
| Completed   |             작업 완료(완료시각 기록) |   X(기본) |         X | 통계 집계 대상           |
| Canceled    |                      취소(사유 필수) |         X |         X | 통계 제외 또는 별도 집계 |

> 기본 운영에서는 Draft 생략 가능(붙여넣기 검증 통과 시 Pending으로 생성).  
> Completed 이후 데이터 변경은 금지하고, 필요 시 **Revision(개정)** 으로 신규 레시피를 생성한다.

### 8.2 상태 전이도

```
                ┌─────────────────────────────────────┐
                │                                     │
[엑셀 붙여넣기] ──→ Draft ──→ Pending ──→ In-Progress ──→ Completed
                  (선택)     (검증통과)    (작업시작)      (완료/LOCK)
                               │             │
                               └──→ Canceled ←┘
                                  (취소/사유필수)
```

- **Draft → Pending**: 검증 통과 시 (Draft 생략 가능)
- **Pending → In-Progress**: 작업 시작 버튼 (작업자 기록)
- **Pending → Canceled**: 등록 취소 (사유 필수)
- **In-Progress → Completed**: 완료 버튼 (2단 확인)
- **In-Progress → Canceled**: 작업 중 취소 (사유 필수)
- **Completed → (변경 불가)**: 오류 발견 시 Revision으로 신규 생성

### 8.3 수정/개정(Revision) 정책

- **Pending**: 내용 수정 가능(단, 수정 이력 Audit Log 기록)
- **In-Progress**: 원칙적으로 수정 금지(필요 시 관리자 승인 + 사유 기록)
- **Completed**: 수정 금지(LOCK)
- **개정(Revision)**: Completed 레시피와 동일한 제품/잉크에 변경이 필요하면
  - 기존 레시피는 유지(증빙)
  - 신규 레시피 생성: `원본ID + revN` 또는 시스템 자동 ID
  - 변경 사유/변경 항목을 로그로 남김

### 8.4 Audit Log(감사로그) 표준 이벤트

| 분류   | 이벤트                   | 필수 기록 필드                                            |
| ------ | ------------------------ | --------------------------------------------------------- |
| 레시피 | 생성(Create)             | recipe_id, 생성자, 생성시각, 원본입력 해시(붙여넣기 원문) |
| 레시피 | 검증 통과/실패(Validate) | 결과(OK/ERROR/WARN), 상세 메시지                          |
| 레시피 | 상태 변경(Status)        | from→to, 수행자, 시각                                     |
| 작업   | 작업 시작(Start)         | 작업자, 시작시각                                          |
| 작업   | 완료(Complete)           | 완료자, 완료시각                                          |
| 작업   | 취소(Cancel)             | 취소자, 사유, 시각                                        |
| 변경   | 수정(Edit)               | 변경 전/후 요약, 사유                                     |
| 개정   | Revision 생성            | 원본ID, 신규ID, 변경 사유                                 |
| 마스터 | 원재료 추가/수정/삭제    | 대상 material_id, 변경 내용, 수행자                       |
| 인증   | 로그인/로그아웃          | user_id, IP, 시각                                         |

> "누가/언제/무엇을"이 남도록 하는 것이 핵심이며, 엑셀 원본보다 시스템 로그가 사후 증빙의 기준이 된다.

## 9. Smart Import 표준화 규칙 (엑셀 붙여넣기 운영 규칙)

기존 Smart Import/Validation 방향은 유지하되, PoC에서 실제 엑셀 데이터 흔들림을 흡수하기 위해 **표준화 규칙**을 명시한다.

### 9.1 헤더 표준화(Header Normalization)

- 공백/특수문자 제거 및 대문자 통일(예: `BYK-199`, `BYK 199`, `byk199` → `BYK-199`로 매핑)
- 동의어(Alias) 매핑 지원(관리자 설정)
  - 예: `GLY`, `GLYCERIN` → `GLYCEROL`(표준명)
- 필수 기본 컬럼(기본 키)
  - `제품명`, `위치(1도/2도/3도)`, `잉크명`, `비고(선택)`

### 9.2 타입/값 파싱 규칙

| 구분             | 파싱 규칙                                    | 예                                   |
| ---------------- | -------------------------------------------- | ------------------------------------ |
| 중량(Weight)     | 숫자만 허용(콤마 제거), 빈값/`-`는 Null 처리 | `1,234.50` → 1234.5                  |
| 횟수/코드(Count) | 문자/코드 허용, 빈값/`-`는 Null 처리         | `APB`, `NPB`, `PITCHY`               |
| 혼합 입력        | 마스터 unit_type에 따라 강제                 | weight 원재료에 문자가 들어오면 오류 |

### 9.3 Validation 단계 정의 (오류 vs 경고)

| 단계          | 정책                                       | 결과                              |
| ------------- | ------------------------------------------ | --------------------------------- |
| Level 1       | 미등록 원재료 포함 시 차단                 | ERROR(등록 불가)                  |
| Level 2       | unit_type 불일치(중량 원료에 문자 등) 차단 | ERROR                             |
| Level 3(선택) | 필수 원료 누락/범위 이탈/금지 조합         | WARN 또는 ERROR(운영 정책에 따라) |

- ERROR: 등록 차단(수정 후 재검증 필요)
- WARN: 등록은 가능하나, 관리자 검토 대상(선택)

### 9.4 엣지 케이스 처리 (추가)

| 상황                           | 처리 방침                                |
| ------------------------------ | ---------------------------------------- |
| 빈 셀                          | Null 처리 (집계 제외)                    |
| 병합 셀 복사                   | 병합 해제 상태로 파싱 시도, 불가 시 WARN |
| 숫자+텍스트 혼합 (예: "1.5kg") | 숫자 부분만 추출 + WARN 표시             |
| 음수 값                        | ERROR (중량/횟수에 음수 불가)            |
| 과도한 값 (이상치)             | WARN (예: 10000Kg 초과 시 확인 요청)     |

### 9.5 Import 원문 보관 정책(권장)

- 사용자가 붙여넣은 **원문 텍스트**는 시스템에 저장(또는 해시 저장)
- 목적: 사후 분쟁/오류 원인 분석("어떤 엑셀을 붙였는지") 증빙

## 10. Work Mode 운영 정책 (Focus Mode를 '절차'로 고도화)

기존 Focus Mode(색상 버튼 클릭 시 해당 색상 컬럼만 표시) UX를 유지하되, "현장 오염 방지" 목적을 강화하기 위해 **작업 절차(체크/로그)** 를 결합한다.

### 10.1 Focus Mode 표준 동작

- Color Focus 활성화 시
  - 해당 색상 원료 컬럼만 노출
  - 그 외 컬럼은 숨김(오염 방지, 집중)
- "전체 보기"로 복귀 가능

### 10.2 작업 시작/완료 정책(권장)

- (선택) 작업 시작 버튼: Pending → In-Progress 전환
  - 작업자는 로그인 사용자로 자동 기록
- 완료 버튼:
  - Completed 전환
  - Work 화면에서는 해당 레시피 행 숨김 처리
  - 통계 집계 대상에 포함

### 10.3 부분 완료 정책 (추가)

- 레시피 내 일부 원재료만 투입 완료된 경우:
  - 상태는 In-Progress 유지
  - 원재료별 체크박스로 투입 완료 표시 (선택 사항)
  - 전체 원재료 투입 완료 시에만 "완료" 버튼 활성화

### 10.4 터치/현장 오작동 방지(권장 UI 정책)

- 완료는 단일 클릭이 아니라 "확인 팝업/길게 누르기/2단 확인" 중 1개 적용 권장
- 완료 후 레시피 내용은 LOCK(수정 불가)

## 11. Master DB(원재료) 관리 정책 고도화 (PoC부터 적용)

현재 마스터 항목(명칭/단위/색상그룹)을 기반으로 하되, PoC 단계부터 최소 필드를 추가하여 Validation 및 확장성을 확보한다.

### 11.1 Master 필드(권장 최소)

| 필드            | 설명                       | 예              |
| --------------- | -------------------------- | --------------- |
| material_id     | 내부 식별자                | 10023           |
| name(표준명)    | 화면/데이터 표준명         | BYK-199         |
| aliases(동의어) | Import 헤더 매핑용         | BYK199, BYK 199 |
| unit_type       | weight / count             | weight          |
| unit(단위)      | g, kg, L, mL, ea           | kg              |
| color_group     | Black/Red/Blue/Yellow/None | None            |
| category(선택)  | 안료/바인더/용제/첨가제    | 첨가제          |
| is_active       | 사용/단종                  | true            |

### 11.2 Master 변경 정책

- 원재료 삭제는 "완전 삭제"보다 **비활성(is_active=false)** 권장(이력 보호)
- Master 변경(추가/수정/비활성)은 Audit Log 필수 기록
- 색상그룹은 초기 4색(Black/Red/Blue/Yellow) 고정 운영, Phase 2에서 사용자 정의 확장 검토

## 12. KPI/운영 지표 (PoC 성공 기준)

PoC(Option A) 단계에서도 효과 검증이 가능하도록 최소 지표를 정의한다.

| 지표              | 정의                                                 | 측정 방법                   | 목표(PoC 1주차 이후 설정) |
| ----------------- | ---------------------------------------------------- | --------------------------- | ------------------------- |
| 입력 리드타임     | 엑셀 붙여넣기→등록 완료까지 시간                     | Management 로그 (자동 산출) | Baseline 대비 50% 단축    |
| 오류 감소         | 미등록 원재료/타입 불일치로 차단된 건수              | Validation 로그             | 오기입 0건                |
| 추적성 완성도     | Completed 중 "작업자/완료시각/로그"가 모두 있는 비율 | Audit Log                   | 100%                      |
| 현장 만족도(정성) | Focus Mode로 작업 동선/오염이 줄었는지               | 사용자 피드백               | 긍정 80% 이상             |

> PoC 1주차에 Baseline을 측정하고, 2주차부터 목표 대비 성과를 추적한다.
