# Attendance Popup Redesign Design

> 2026-04-25 근태 알림 팝업 리디자인 설계

## 1. Overview

| Item | Detail |
|------|--------|
| Feature | attendance-popup-redesign |
| Target | `tray_client/src/attendance_popup.py` |
| Context | Tkinter 기반 고정 팝업, 우하단 표시, 확인 전까지 유지 |
| Goal | 기존 팝업을 더 세련되고 덜 부담스러운 업무용 알림 패널로 개선 |

이 설계는 트레이 클라이언트의 근태 알림 팝업만 다룬다. 서버 API, 알림 판단 기준, 개인정보 보호 정책 자체는 바꾸지 않는다.

## 2. Goals

1. Windows 업무 도구에 자연스럽게 섞이는 미니멀한 팝업으로 보이게 한다.
2. 알림 내용을 읽기 쉽게 정리하되, 지각/조퇴 같은 상세 사유는 노출하지 않는다.
3. `이름 3명까지 + 외 N명` 규칙으로 정보량을 고정해 창이 불필요하게 커지지 않게 한다.
4. `확인`, `닫기`, `오늘은 그만` 동작을 시각적으로 명확히 구분한다.
5. 기존 동작인 "사용자 액션 전까지 유지"와 "중복 팝업 누적 방지"를 유지한다.

## 3. Non-goals

1. Windows 네이티브 토스트로 복귀하지 않는다.
2. 팝업에서 사번, 부서, 근태 사유, 이상 종류를 직접 노출하지 않는다.
3. 알림 소리, 서버 폴링 주기, 중복 억제 시그니처 계산 방식은 이번 범위에서 바꾸지 않는다.
4. 팝업 클릭 드래그, 리사이즈, 복수 창 스택 같은 상호작용은 추가하지 않는다.

## 4. Current Constraints

현재 구현은 `tkinter.Toplevel` 한 개를 재사용하는 구조다.

- 장점:
  - 사용자 확인 전까지 유지 가능
  - 새 알림이 와도 기존 창 내용만 갱신 가능
  - 빌드/배포 복잡도가 낮음
- 제약:
  - HTML/CSS 수준의 자유로운 레이아웃은 어렵다
  - 버튼/텍스트 스타일은 Tkinter 위젯 범위 안에서 정리해야 한다
  - 그림자, 둥근 모서리, 반투명 효과는 제한적이다

설계는 이 제약을 인정하고, Tkinter 안에서 최대한 정돈된 결과를 목표로 한다.

## 5. Visual Design

### 5.1 Tone

- 방향: 미니멀형, Windows 업무 패널에 가까운 톤
- 인상: 조용하지만 선명함, 과한 경고창 느낌 금지
- 색 사용: 전체 경고 배경색 금지, 상단 포인트와 버튼 강조 정도만 허용

### 5.2 Window Layout

- 위치: 화면 오른쪽 아래, 작업표시줄 위로 20~24px 정도 띄움
- 폭: 360~380px 고정
- 높이: 내용에 맞춰 자동
- 배경: 흰색
- 외곽선: 1px 연회색
- 그림자: 약하게 적용
- 모서리: Tkinter 한계 내에서 최대한 부드럽게 보이도록 내부 여백과 레이어로 정리

### 5.3 Content Layout

상단, 본문, 버튼의 3단 구조로 고정한다.

1. 상단 헤더
   - 작은 상태 점
   - 제목: `근태 확인 필요`
   - 우측 배지: `N명`

2. 보조 문구
   - 짧은 설명 1줄
   - 예: `오늘 확인이 필요한 인원이 있습니다.`

3. 본문 목록
   - 이름 3명까지 개별 줄 표시
   - 각 줄 문구: `{이름} 인원 근태 확인`
   - 4명 이상이면 마지막 줄에 `외 N명`

4. 하단 버튼
   - `확인`: 채운 버튼, 주요 액션
   - `닫기`: 보조 버튼
   - `오늘은 그만`: 가장 약한 보조 버튼

## 6. Content Rules

### 6.1 Privacy

팝업에는 아래 정보만 표시한다.

- 이름
- 총 인원 수
- 일반화된 행동 문구 (`인원 근태 확인`)

표시하지 않는 정보:

- 지각, 조퇴, 외출, 누락 등 상세 사유
- 사번
- 부서
- 근태코드
- 날짜별 상세 내역

### 6.2 Summary Text

라이브 알림 기본 규칙:

- 제목: `근태 확인 필요`
- 배지: `N명`
- 보조 문구: `오늘 확인이 필요한 인원이 있습니다.`

테스트 알림 규칙:

- 제목: `근태 알림 테스트`
- 배지: `TEST`
- 보조 문구: `팝업 표시와 버튼 동작을 확인하세요.`

## 7. Interaction Design

### 7.1 Show / Update

- 팝업은 동시에 하나만 유지한다.
- 새 알림이 오면 새 창을 띄우지 않고 기존 창 내용을 갱신한다.
- 중복 시그니처가 같으면 기존 억제 정책을 그대로 사용한다.

### 7.2 Buttons

- `확인`
  - 팝업 닫기
  - 근태 페이지 열기
- `닫기`
  - 현재 팝업만 닫기
  - 오늘 알림 비활성화는 하지 않음
- `오늘은 그만`
  - 현재 팝업 닫기
  - 오늘 자정 전까지 근태 팝업 비활성화

### 7.3 Empty / Overflow

- 대상 인원이 0명이면 팝업을 띄우지 않는다.
- 1~3명은 모두 표시한다.
- 4명 이상이면 3명 표시 후 `외 N명`

## 8. Component Changes

### 8.1 `tray_client/src/attendance_popup.py`

주요 변경 대상.

- `PopupPayload`를 헤더/배지/보조 문구/목록/보조 상태까지 표현할 수 있게 확장
- 기존 단순 `Label + Button` 나열 구조를 카드형 레이아웃으로 재구성
- 본문 줄 렌더링을 "최대 3명 + 외 N명" 규칙으로 변경
- 버튼 스타일 분리
- 창 크기/여백/위치 계산 재조정

### 8.2 `tray_client/src/attendance_alerts.py`

- 라이브/테스트 payload 생성 함수가 새 payload 구조를 따르도록 조정
- 라이브 팝업 요약 문구를 날짜/근무구분 중심이 아닌 일반형 안내 문구로 단순화

### 8.3 `tray_client/src/main.py`

- 동작 자체는 유지
- `확인`, `닫기`, `오늘은 그만`이 새 UI와 같은 의미로 연결되는지만 확인

## 9. Error Handling

1. Tkinter 사용 불가
   - 기존처럼 로그만 남기고 팝업 생성을 건너뜀

2. 브라우저 열기 실패
   - 기존처럼 warning 로그 남김
   - 팝업 자체는 비정상 종료하지 않음

3. payload 비정상
   - 이름이 비면 `확인 필요`
   - 목록이 비면 일반 안내 1줄 표시

## 10. Testing

### 10.1 Unit Tests

- payload 생성:
  - 1명, 3명, 4명 이상 표시 규칙
  - 테스트용 payload와 라이브 payload 문구 구분
- privacy 규칙:
  - 상세 사유가 payload 문구에 들어가지 않는지 확인
- 액션:
  - `확인`은 페이지 열기
  - `오늘은 그만`은 당일 mute 처리

### 10.2 Manual Checks

1. 우하단 위치가 작업표시줄과 겹치지 않는지
2. 제목, 배지, 이름 목록이 좁은 해상도에서도 줄바꿈 없이 안정적인지
3. 4명 이상일 때 `외 N명`이 자연스럽게 보이는지
4. `확인` 버튼이 가장 먼저 눈에 들어오는지
5. 테스트 알림과 실알림이 같은 레이아웃으로 보이는지

## 11. Rollout

1. 팝업 UI 리디자인 구현
2. 테스트 갱신
3. 로컬 빌드 및 설치 파일 재생성
4. 실제 트레이 실행 후 위치/가독성 점검

## 12. Scope Check

이번 설계는 팝업 UI와 텍스트 구성만 다룬다. 알림 판단 로직, 서버 응답 포맷, 근태 상세 화면 구조는 범위 밖이다.
